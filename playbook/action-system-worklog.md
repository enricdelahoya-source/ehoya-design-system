# Worklog — 2026-04-13
## State-driven action system

## Context

Continued work on the AI panel and action model for the case management prototype.

The goal was to move from generic “recommended actions” to a structured, state-driven decision system.

---

## 1. Shift from recommendations to decisions

Replaced:
- “Recommended actions”

With:
- “Next step”
- one primary action
- supporting explanation
- secondary actions

This introduced:
- clear hierarchy
- reduced ambiguity
- more operational tone

---

## 2. Introduced state-driven model

Extracted action logic into a dedicated system layer:

- state → defines available actions
- primary + secondary actions per state

Removed decision logic from UI components.

---

## 3. Defined action vocabulary

Created a fixed set of system actions:
- no AI-generated actions
- all actions map to real operations

This reinforced:
- consistency
- trust
- predictability

---

## 4. Added explanation layer

Separated:
- decision logic (system)
- explanation (AI / text layer)

Key idea:
> system decides, AI explains

---

## 5. Added multiple scenarios

Created realistic mock cases to validate the system:

- pending approval
- waiting on customer
- ready to resolve
- escalated
- escalated with SLA risk

Validated that:
- the same UI pattern works across all cases
- actions adapt correctly to state

---

## 6. Introduced signal-based override

Added first signal:
- SLA risk

Behavior:
- escalated + high SLA risk → notify stakeholder
- otherwise → route to specialist

Important constraint:
- signals only reorder actions
- they do not introduce new ones

---

## 7. Fixed action duplication

Issue:
- primary action also appeared in secondary list

Fix:
- filtered secondary actions to remove primary

Result:
- cleaner hierarchy
- clearer decision model

---

## 8. Aligned state and action (CASE-10526)

Issue:
- state: "In investigation"
- action: "Resolve case"

Decision:
- update state to "Ready to resolve"

Result:
- consistent mental model
- clearer UI

---

## 9. Added edge cases

### A. Conflicting signals
- pending approval + high SLA risk
- primary remains approval
- explanation reflects urgency

### B. No-action state
- waiting on customer + noActionAvailable
- no primary action rendered
- explanation communicates waiting state

This introduced:
- constraint handling
- system restraint (no fake actions)

---

## 10. UI behavior refinement

Handled case with no primary action:
- removed button
- preserved layout structure
- ensured spacing still feels intentional

---

## 11. Validation

Confirmed correct behavior for:

- CASE-10521 → pending approval → route to approval
- CASE-10524 → waiting on customer → send follow-up
- CASE-10526 → ready to resolve → resolve case
- CASE-10504 → escalated → route to specialist
- CASE-10527 → escalated + high SLA → notify stakeholder
- CASE-10531 → pending approval + high SLA → route to approval
- CASE-10532 → waiting on customer (no action) → no primary

---

## Key learnings

- actions must be grounded in system state, not generated by AI
- one primary action simplifies decision-making significantly
- signals should refine, not replace, state logic
- constraints (like approval) should override urgency
- showing no action is better than showing a bad action

---

## Outcome

The system evolved from:
- a recommendation UI

to:
- a structured decision system with:
  - state model
  - action model
  - signal overrides
  - edge case handling
